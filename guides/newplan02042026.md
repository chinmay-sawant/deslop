# Master Roadmap For deslop Next Phase

## Summary

- Target document: [guides/master-roadmap.md](/home/chinmay/ChinmayPersonalProjects/deslop-codex/guides/master-roadmap.md).
- Near-term direction: deepen Go and Python, harden evaluation and docs, keep Rust intentionally narrower, and defer a new backend until the product surface is easier to govern.
- Current baseline is healthy: all current tests passed via `cargo test --locked` on April 2, 2026.
- Current rule surface is already large enough that governance now matters as much as raw count: the feature guide documents roughly 270 rule IDs, but roadmap docs are fragmented, several status documents are stale, and the repo repeatedly references a missing `implementation-guide.md`.
- Treat `500 rules` as a medium-term inventory target across `stable`, `experimental`, and `research` buckets, not as the next default-on release target.

## Implementation Changes

1. **Phase 0 - Create one source of truth before adding much more surface area.** Build the master guide under `guides/`; resolve the missing implementation-guide reference; normalize or archive stale `advanceplan*` documents; add a simple guide taxonomy such as `active`, `completed`, and `backlog`; explicitly mark which historical plans are shipped versus still actionable.
2. **Phase 0 - Add a rule metadata registry in code.** Centralize rule metadata as `id`, `language`, `family`, `default_severity`, `status`, `configurability`, and one-line description; stop maintaining rule inventories manually across README, guides, and frontend content.
3. **Phase 0 - Add a machine-readable rule listing surface.** Introduce `deslop rules --json`, `deslop rules --language <lang>`, and optionally `deslop rules --status <stable|experimental|research>`; keep existing scan output backward-compatible and make the new listing additive.
4. **Phase 0 - Add docs-sync automation.** Generate or validate README rule tables, frontend docs content, GitHub Action examples, and the feature guide from the rule registry; add a CI check that fails when action inputs, README examples, or docs drift from the code.
5. **Phase 1 - Formalize evaluation and promotion policy.** Turn the existing real-repo comparison workflow into a first-class corpus harness; keep `gopdfsuit` and `SnapBack`, add a small representative set of Go services, Python web apps, Python ML repos, and Rust applications; record per-rule false-positive notes before severity promotion.
6. **Phase 1 - Add rule stability tiers.** Define `stable` for default-on low-noise rules, `experimental` for opt-in packs, and `research` for documented backlog ideas; require every new rule to declare its tier up front.
7. **Phase 1 - Set a per-rule merge contract.** Every new detector must ship with one positive fixture, one negative fixture, suppression/config coverage when applicable, one mixed-language regression if the parser or index changes, and at least one real-repo validation note before graduating to `stable`.
8. **Phase 2 - Expand Go where the repo is currently most skewed.** Prioritize framework and service gaps that are not yet broad enough beyond Gin and GORM: `net/http`, `chi`, `echo`, `fiber`, `grpc`, `sqlc`, `ent`, `bun`, `pgxpool`, `redis`, upstream batching and retry/backoff, streaming/export inefficiencies, config and environment misuse, request-path setup churn, and broader request-boundary reliability signals.
9. **Phase 2 - Keep Go growth application-oriented.** Bias new Go rules toward HTTP services, data access, concurrency, streaming, and operational correctness; avoid spending the next milestone on taste-level style rules; target roughly 220 stable Go rules before greenlighting a fourth backend.
10. **Phase 3 - Expand Python on framework, packaging, and boundary-awareness.** Prioritize Django REST Framework, Celery, asyncio/httpx/aiohttp, SQLModel, Pydantic v2, click/typer CLI boundaries, local module graph resolution, `pyproject.toml` awareness, broader repo-level API-shape checks, and better public-contract precision beyond `Any`.
11. **Phase 3 - Expand Python on AI/ML and LLM ecosystems selectively.** Extend the current ML/LLM pack into LangChain/LlamaIndex-style orchestration, vector database client misuse, embedding batch/cache patterns, tokenizer/model lifecycle reuse, and more robust prompt/request boundary rules; keep all of these import-gated so non-ML repos stay quiet.
12. **Phase 3 - Treat Pythonâ€™s remaining partials as explicit epics.** Create issue families for import-time lifecycle, environment/config bootstrapping, retry/timeouts, schema validation at external boundaries, broader framework-specific N+1 or over-fetching only when query evidence is strong, and repository-level duplication refinement.
13. **Phase 4 - Grow Rust narrowly and deliberately.** Focus Rust on cargo workspace awareness, crate-graph and trait/impl-local resolution, macro-aware escapes, async runtime families (`tokio`, `axum`, `actix`, `tonic`), allocation and clone waste, config/builder misuse, and application-boundary rules; do not try to force Rust into parity with Go or Python by count.
14. **Phase 4 - Set a smaller Rust target.** Aim for roughly 60 to 90 high-signal Rust rules centered on application code and local correctness review, not a huge generic lint surface.
15. **Phase 5 - Only then start a new backend.** First new language should be TypeScript/JavaScript because the current architecture and market fit strongly favor it; second should be Java for Spring/JPA-style enterprise service patterns; third only if demand is clear should be C#; explicitly defer Kotlin until Java exists, and defer PHP, Ruby, and C/C++ unless a real evaluation corpus justifies them.
16. **Issue backlog to create immediately.** `master-roadmap-guide`, `missing-implementation-guide-resolution`, `rule-registry-and-rules-cli`, `docs-sync-ci`, `real-repo-corpus-harness`, `rule-stability-tiers`, `go-web-framework-expansion`, `go-data-layer-expansion`, `python-framework-expansion`, `python-module-graph-and-packaging-awareness`, `python-ml-llm-ecosystem-pack`, `rust-workspace-runtime-pack`, and `typescript-backend-research-spike`.

## Public APIs / Interfaces / Types

- Add a central rule metadata structure that is authoritative for rule inventory, docs generation, and the future `rules` CLI.
- Add `deslop rules` as a new additive CLI surface with JSON output and filtering by language and status.
- Keep `scan`, `bench`, current JSON report shape, and current config behavior backward-compatible.
- If experimental packs become configurable, add them as additive config keys rather than changing existing rule IDs or suppression behavior.

## Test Plan

- Preserve the current green baseline with `cargo test --locked` as the non-negotiable gate.
- Add CI coverage for rule-registry consistency, docs-sync consistency, and action-surface consistency.
- For every new rule family, require positive fixture, negative fixture, suppression/config fixture when relevant, and real-repo comparison evidence.
- For every parser or index expansion, require a mixed-language regression test and a benchmark note update if scan cost changes materially.
- Keep benchmark and finding-diff notes as first-class artifacts, not ad hoc reports.

## Assumptions And Defaults

- Use one master guide, not another wave of loosely connected per-language plan files.
- Prioritize Go/Python depth and product hardening before any new backend implementation.
- Use `500` as the medium-term roadmap inventory target, with a smaller stable default-on target for each release.
- Expect the master guide to include a clear separation between `stable now`, `experimental next`, and `research/deferred`.
